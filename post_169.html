  2. transport

  3. network

  4. link
    wifi, lte, ethernet 등
  5. physical

네트워크 앱 계층

  • email
  • web
  • game
  • chatting
  • share file
  • 등과 같은 프로그램에 쓰이는 계층.
  • 결국 프로세스.
  • sw 개발할 때 신경써야하는 부분.

클라이언트-서버 구조

  • 서버는 고정된 IP 를 가진다.
  • 클라이언트는 고정되지 않아도 된다.
  • 서버의 프로세스와 클라이언트의 프로세스간 통신.
  • 이 통신할 수 있는 인터페이스를 socket 이라고 한다.
  • 즉 socket 은 app 계층에서 사용되는 것이다.
  • 이 socket 을 사용하기 위해서는 IP 주소(어떤 컴퓨터)와 port(어떤 프로세스) 가 필요하다.
  • 한 장치에는 여러가지 프로세스가 돌아가고 있다.
  • 네트워크 앱도 하나의 프로세스이므로 특정 port 가 필요한 것이다!

웹 서비스 포트

  • 거의 모든 웹 서비스는 포트번호가 80 으로 고정.
  • 동일한 프로토콜을 사용하기 때문에
  • 또한 웹 서버라는 것은 언제나 고정된 위치에 있어야 한다는 특성 때문에
  • IP 뿐 아니라 웹 서버 sw 가 돌아가는 프로세스를 대변하는 port 도 고정하기 위함.

app 계층 요구사항

  • 결국 하위 계층인 transport 계층의 서비스를 받는다.
    app 계층에서 사용할 수 있게 그 하위 계층에게 요구하는 요소
  1. data integrity
    데이터 유실되지 않고 신뢰성 있어야 함
  2. timing
    지연시간이 적어야함
    빨리 전송 되는 것.
  3. throughput
    처리량이 좋아야 함
    단위 시간당 많이 받는 것.
  4. security
  • TCP 는 이 중에서 1 번 항목만 지원해주고 있다!
  • UDP 는 1 번도 제공 안 한다.
  • 보안도 app 계층에서 처리해야 한다.

HTTP

  • HyperText Transfer Protocol
  • 결국 웹 서버에 문서를 요청하고 응답하는 것으로 매우 단순함.
  • 하위 계층인 TCP 계층을 사용함.
  • TCP 커넥션을 먼저 하게 된다.
  • stateless, 무상태 프로토콜임.
    상대방에 대한 정보를 기억하지 않는다.

HTTP connection

  • TCP connection 에 따라 2 가지로 나뉨
  • non persistence HTTP
    TCP 커넥션이 끊어짐
  • persistent HTTP
    TCP 커넥션이 유지됨
    웹 서비스는 persistent HTTP
    그리고 파이프라인 개념을 이용하여 더 효율적으로 데이터를 전달한다.
